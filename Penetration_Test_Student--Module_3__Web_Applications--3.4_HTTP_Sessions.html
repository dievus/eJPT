3.4 HTTP Sessions


HTTP Sessions



• Occasionally developers prefer to store information on the server instead of client
◇ happens to hide the application logic, or just avoid the back and forth data transmission typical of cookies
• Sessions allow websites to store variables specific for a given visit on the server side
• Each session is identified by a session id, or token, which the server assigns to a client
• Client then presents this ID for each subsequent request to be recognized by the server
• Using the session ID, server retrieves the state of the client and all variables associated with it
◇ Server stores session ID's inside text files in its storage.


Session Cookies



• Session cookies contain a single parameter value pair referring to the session



• Websites install session cookies in different ways
◇ Websites running PHP install session cookies utilizing “PHPSESSID”
◇ JSP websites install session cookies utilizing “JSESSIONID”
◇ Each dev language has its own default session parameter name
• If needed, servers install session cookies after a browser performs an activity, such as:
◇ Opening a page
◇ Changing web application settings
◇ Logging in
• Browser then used the cookie in follow on requests
◇ Session could contain many variables, so sending a small cookie keeps bandwidth low

Session Cookie Example



• Client uses a login form to POST the user's credentials



• Server sends back a response with a Set-cookie header field
◇ Contains the session ID



• Browser sends back the cookie according to the cookie protocol, sending the session ID



GET Requests



• Session IDs can be transmitted via GET requests as well
